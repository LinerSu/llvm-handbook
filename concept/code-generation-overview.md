# Code Generation (Backend) Overview

> 🧭 **Concept** · `concept · codegen · llvm` · Index [[LLVM.MOC]]
> **Prerequisites:** [[llvm-basics]], [[ssa-form]] · **Tool:** `llc`

> [!abstract] Chapter map
> How the middle-end's optimized LLVM IR becomes target machine code: a sequence of phases over **Machine IR (MIR)** — instruction selection → scheduling → register allocation → emission — driven by target descriptions generated from TableGen.

> [!info]+ From classic compiler theory → LLVM
> | Classic concept | LLVM realization |
> |---|---|
> | Instruction selection (IR → target ops) | **SelectionDAG** (default), **GlobalISel** (newer), **FastISel** (`-O0`) |
> | Target machine code IR | **MIR**: `MachineFunction` → `MachineBasicBlock` → `MachineInstr` |
> | Register allocation (∞ virtual → finite physical) | Greedy / Basic / Fast / PBQP allocators |
> | Assembler / object writer | the **MC** layer (`MCInst`, `MCStreamer`) |
> | Target description tables | generated by **TableGen** from `*.td` files in `lib/Target/<Arch>` |

---

### 1. Definition

> [!note] Definition
> The **backend** (target-independent code generator + a target description) lowers optimized LLVM IR into native code. It works on **MIR**, a lower-level SSA-then-non-SSA representation of target instructions with *virtual* registers, progressively constraining the program until it is concrete machine code. The standalone driver is `llc`.

### 2. The pipeline (phases)

> [!info] In order
> 1. **Instruction selection** — map IR operations to target `MachineInstr`s. **SelectionDAG** builds a per-block DAG and pattern-matches; **GlobalISel** is a newer function-wide pipeline (IRTranslator → Legalizer → RegBankSelect → InstructionSelect) on generic MIR; **FastISel** trades quality for speed at `-O0`. → deep dive: [[instruction-selection]].
> 2. **Instruction scheduling** — order instructions to hide latency / exploit instruction-level parallelism (pre- and post-register-allocation schedulers). → deep dive: [[instruction-scheduling]].
> 3. **SSA-based machine optimizations** — peephole, machine CSE, etc., while MIR virtual registers are still in SSA.
> 4. **Register allocation** — map virtual registers to physical registers, inserting **spill** code when they don't fit. This **eliminates φ** (takes MIR out of SSA). Allocators: **Greedy** (default at `-O1+`), **Basic**, **Fast** (`-O0`), **PBQP**. → deep dive: [[register-allocation]].
> 5. **Prologue/epilogue insertion** — finalize the stack frame (saved registers, frame pointer, stack adjustment). → [[shrink-wrapping]] moves these to the smallest region that needs the frame.
> 6. **Late / post-RA passes** — post-RA scheduling, peephole, branch folding.
> 7. **Code emission** — lower `MachineInstr` → `MCInst` and emit assembly (`.s`) or an object file (`.o`) through the **MC** layer.

### 3. Key data structures

> [!info] MIR and target hooks
> `MachineFunction` / `MachineBasicBlock` / `MachineInstr`; `MachineRegisterInfo` (virtual vs physical regs); and the target interfaces `TargetLowering` (how IR is lowered), `TargetInstrInfo`, `TargetRegisterInfo`. Most of each target's register/instruction/calling-convention/ISel-pattern tables are **generated by TableGen** from `lib/Target/<Arch>/*.td`.

### 4. Where it sits

`llvm/lib/CodeGen/` is target-independent; `llvm/lib/Target/<Arch>/` is the per-ISA description; `llvm/lib/MC/` is the machine-code/object layer. (Crosswalk in [[LLVM.MOC]].)

> [!warning] Version-sensitive specifics — confirm against current docs
> The **default ISel framework and register allocator evolve**: SelectionDAG remains the default on many targets while **GlobalISel** is most mature on AArch64 and is being adopted elsewhere; **Greedy** is the default allocator at `-O1+`. Treat these defaults as of the version this vault tracks ([[llvm-version]]) and re-check [CodeGenerator.html](https://llvm.org/docs/CodeGenerator.html) for the exact version you use. *(This note's pipeline structure is stable; the defaults are the moving part.)*
